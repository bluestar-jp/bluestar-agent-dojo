# Requirements Clarification Guide

- Purpose: 要件明確化技法の詳細ガイド
- Scope: テスト可能な要件の品質基準と明確化手法

## テスト可能な要件の特徴

IEEE 830に基づく良い要件の特性:

1. **具体的（Specific）**: 曖昧さのない明確な記述
2. **測定可能（Measurable）**: 定量的に検証可能
3. **達成可能（Achievable）**: 技術的に実現可能
4. **追跡可能（Traceable）**: 上位要件や設計との対応が明確
5. **テスト可能（Testable）**: テストケースを導出できる

## よくある曖昧パターン

| パターン | 例 | 明確化方法 |
| --- | --- | --- |
| 主観的表現 | 「使いやすい」 | 具体的なユーザビリティ指標を定義 |
| 未定量表現 | 「高速に」 | レスポンスタイムの数値基準を設定 |
| 範囲未定義 | 「テキスト入力」 | 文字数、文字種、フォーマットを定義 |
| 条件の欠如 | 「データを保存する」 | 成功/失敗条件、排他制御を明示 |
| 暗黙の前提 | 「ログイン済みユーザー」 | 認証状態、権限レベルを明示 |

## 質問テンプレート

### 入力ドメインの明確化

- このフィールドの有効な値の範囲は？
- 許容される文字種は？（英数字のみ、マルチバイト文字含む等）
- 最大/最小の長さ・値は？
- null/空文字は許容されるか？

### 振る舞いの明確化

- 正常系の期待結果は？
- 無効な入力時のエラーメッセージは？
- タイムアウト時の振る舞いは？
- 同時実行時の排他制御は？

### 非機能要件の明確化

- 許容されるレスポンスタイムは？
- 同時接続数の上限は？
- データの保持期間は？
